> 本文档基于 Google Code Review 指南整理，适用于所有技术栈项目的代码评审规范。

---

## 一、核心原则

1. **追求持续改进，而非完美** — 倾向于批准能提升代码健康状态的变更，不要求每次都达到完美
2. **对事不对人** — 评论代码问题，而不是评论开发者；使用建设性的语言
3. **解释为什么** — 每条评审意见都应说明原因，帮助开发者理解并成长
4. **提供建议** — 指出问题的同时给出具体的改进方案

---

## 二、问题分级

| 前缀 | 含义 | 处理方式 |
|------|------|---------|
| `[必须]` | 严重问题，影响正确性/安全性/可维护性 | 阻止合入，必须修复 |
| `[建议]` | 可改进但不阻塞的问题 | 讨论后决定是否修改 |
| `[Nit]` | 小问题，如命名、格式等 | 可忽略，作者自行决定 |

## 三、检查维度与规则清单

### 3.1 设计与架构

| 规则 ID | 描述 | 优先级 |
|---------|------|--------|
| DESIGN-001 | 代码是否属于正确的模块/库 | 高 |
| DESIGN-002 | 是否能与系统其他部分良好集成 | 高 |
| DESIGN-003 | 现在是否是添加此功能的合适时机 | 中 |
| DESIGN-004 | 是否存在不必要的复杂设计（过度工程） | 高 |

**检查要点：**
- CL 中各种代码的交互是否有意义
- 此变更是属于代码库还是应该属于独立库
- 现在是添加此功能的好时机吗

### 3.2 功能与正确性

| 规则 ID | 描述 | 优先级 |
|---------|------|--------|
| FUNC-001 | 代码行为是否符合预期 | 高 |
| FUNC-002 | 是否处理边缘情况和异常 | 高 |
| FUNC-003 | 并发操作是否安全 | 高 |
| FUNC-004 | 对用户是否友好 | 中 |

**检查要点：**
- 代码的行为是否符合作者的预期
- 考虑边缘情况、并发问题
- 对于 UI 变更，验证变化的合理性

### 3.3 复杂度与可维护性

| 规则 ID | 描述 | 优先级 |
|---------|------|--------|
| COMP-001 | 代码是否可以被简化 | 中 |
| COMP-002 | 其他开发人员能否轻松理解 | 高 |
| COMP-003 | 是否实现了不确定的未来功能 | 中 |

**检查要点：**
- 代码是否可以被简化
- 其他开发人员是否能够轻松理解和使用该代码
- 警惕开发者实现未来可能需要但现在不确定的功能

### 3.4 测试

| 规则 ID | 描述 | 优先级 |
|---------|------|--------|
| TEST-001 | 是否包含自动化测试 | 高 |
| TEST-002 | 测试是否能在代码破坏时失败 | 高 |
| TEST-003 | 测试是否设计良好 | 中 |
| TEST-004 | 测试是否会产生误报 | 中 |

**检查要点：**
- 代码中是否包含正确且设计良好的自动化测试
- 测试是否真的会在代码被破坏时失败
- 每个测试是否有简单而有用的断言

### 3.5 命名与注释

| 规则 ID | 描述 | 优先级 |
|---------|------|--------|
| NAME-001 | 变量/类/方法名是否清晰 | 高 |
| NAME-002 | 名称长度是否合适 | 中 |
| COMMENT-001 | 注释是否清晰有用 | 中 |
| COMMENT-002 | 注释是否解释"为什么"而非"做什么" | 中 |

**检查要点：**
- 名称应该足够长以完全传达内容或作用，但不会太长难以阅读
- 注释应该解释"为什么"而不是"做什么"

### 3.6 代码规范

| 规则 ID | 描述 | 优先级 |
|---------|------|--------|
| STYLE-001 | 是否遵循团队/项目代码规范 | 中 |
| STYLE-002 | 代码整体风格是否一致 | 中 |
| DOC-001 | 相关文档是否同步更新 | 建议 |

**JavaScript/TypeScript 必须规则：**

| 规则 | 说明 | 示例 |
|------|------|------|
| 使用 `const` 定义引用 | 避免使用 `var` | `const name = 'John';` |
| 重新赋值时使用 `let` | 代替 `var` | `let count = 0;` |
| 使用字面量语法 | 创建对象和数组 | `const obj = {};` |
| 使用字符串模板 | 代替字符串拼接 | `` `Hello, ${name}` `` |
| 永远不要使用 `eval()` | 安全风险 | — |
| 使用 `===` 和 `!==` | 而不是 `==` 和 `!=` | `if (a === b)` |
| 多行代码块使用大括号 | 包裹代码块 | `if (x) { ... }` |
| 语句结束加分号 | 避免 ASI 陷阱 | `const x = 1;` |

### 3.8 安全审计（OWASP Top 10 前端相关）

| 规则 ID | 描述 | 优先级 | OWASP |
|---------|------|--------|-------|
| SEC-001 | 是否存在跨站脚本攻击风险（innerHTML、v-html、dangerouslySetInnerHTML） | 高 | A03 |
| SEC-002 | 用户输入是否经过校验和转义处理 | 高 | A03 |
| SEC-003 | 动态 URL 是否经过校验（禁止 javascript:、data: 协议） | 高 | A03 |
| SEC-004 | 敏感信息是否存储在 localStorage/sessionStorage | 高 | A02 |
| SEC-005 | JWT/Token 是否安全存储和传输 | 高 | A07 |
| SEC-006 | 是否仅依赖前端进行权限控制 | 高 | A01 |
| SEC-007 | 敏感操作是否有 CSRF Token 保护 | 高 | A01 |
| SEC-008 | 第三方依赖是否存在已知漏洞 | 高 | A06 |
| SEC-009 | 外部 CDN 资源是否使用 SRI 校验 | 中 | A08 |
| SEC-010 | 是否在前端暴露敏感配置或 API 密钥 | 高 | A02 |
| SEC-011 | 是否考虑 X-Frame-Options 或 CSP frame-ancestors | 中 | A05 |
| SEC-012 | URL 跳转是否校验目标地址白名单 | 高 | A01 |

**关键安全规则详解：**

**A03 - 注入（XSS 防护）**

```javascript
// 禁止：XSS 风险
element.innerHTML = userInput;
// Vue
<div v-html="userContent"></div>
// React
<div dangerouslySetInnerHTML={{ __html: userContent }} />

// 正确：使用 textContent 或框架自动转义
element.textContent = userInput;
<div>{{ userContent }}</div>  // Vue 自动转义
<div>{userContent}</div>      // React 自动转义
```

**A02 - 加密失败**
- 敏感数据（密码、Token）不应存储在 localStorage
- 避免在前端硬编码 API 密钥、加密密钥
- 确保使用 HTTPS 传输敏感信息

**A07 - 认证失败**
- Token 应存储在 httpOnly Cookie 中（而非 localStorage）
- 实现 Token 刷新机制
- 敏感操作需要二次验证

**A08 - 软件完整性失败（SRI 校验）**

```html
<!-- 正确：使用 SRI 校验外部 CDN 资源 -->
<script src="https://cdn.example.com/lib.js"
        integrity="sha384-xxxx"
        crossorigin="anonymous"></script>
```

### 3.9 性能检查

| 规则 ID | 描述 | 优先级 |
|---------|------|--------|
| PERF-001 | 事件监听器、定时器、订阅是否正确清理 | 高 |
| PERF-002 | 长列表是否使用虚拟滚动 | 高 |
| PERF-003 | 组件是否存在不必要的重复渲染 | 中 |
| PERF-004 | 图片是否懒加载、是否使用合适的格式和尺寸 | 中 |
| PERF-005 | 是否按路由/功能进行代码分割 | 中 |
| PERF-006 | 是否有重复请求、是否使用缓存和防抖 | 中 |
| PERF-007 | 复杂计算是否使用 useMemo/computed 缓存 | 中 |
| PERF-008 | 是否引入了过大的依赖包 | 中 |
| PERF-009 | 是否存在频繁的 DOM 操作导致重排重绘 | 中 |
| PERF-010 | 非关键资源是否异步/延迟加载 | 中 |

**关键性能规则详解：**

**PERF-001 内存泄漏**

```javascript
// 错误：未清理
useEffect(() => {
  window.addEventListener('resize', handleResize);
  const timer = setInterval(poll, 1000);
}, []);

// 正确：返回清理函数
useEffect(() => {
  window.addEventListener('resize', handleResize);
  const timer = setInterval(poll, 1000);
  return () => {
    window.removeEventListener('resize', handleResize);
    clearInterval(timer);
  };
}, []);
```

```vue
<!-- Vue 3 示例 -->
<script setup>
import { onMounted, onUnmounted } from 'vue';
let timer;
onMounted(() => { timer = setInterval(poll, 1000); });
onUnmounted(() => { clearInterval(timer); });
</script>
```

**PERF-002 大列表渲染**
- 超过 100 条数据的列表考虑虚拟滚动
- 推荐：`vue-virtual-scroller`、`react-window`、`@tanstack/virtual`

**PERF-003 不必要渲染**

```javascript
// 错误：每次渲染都重新计算
function Component({ items }) {
  const sorted = items.sort((a, b) => a.name.localeCompare(b.name));
  return <List items={sorted} />;
}

// 正确：缓存计算结果
function Component({ items }) {
  const sorted = useMemo(
    () => [...items].sort((a, b) => a.name.localeCompare(b.name)),
    [items]
  );
  return <List items={sorted} />;
}
```

**PERF-005 代码分割**

```javascript
// 路由懒加载
const UserProfile = () => import('./views/UserProfile.vue');
const Dashboard = React.lazy(() => import('./pages/Dashboard'));
```

**PERF-006 网络请求优化**

```javascript
// 搜索防抖
const debouncedSearch = useDebouncedCallback(
  (value) => fetchResults(value),
  300
);
```

**PERF-008 Bundle 体积**

```javascript
// 错误：引入整个 lodash（约 70KB）
import _ from 'lodash';

// 正确：按需引入（约 2KB）
import debounce from 'lodash-es/debounce';
```

## 四、代码质量评分标准

### 基础分（80分）

**代码可读性（30分）**

| 检查项 | 分值 | 说明 |
|--------|------|------|
| 符合代码规范 | 8 | 遵循团队/公司代码规范 |
| 变量命名清晰 | 8 | 变量名、方法名、类名有较高可读性 |
| 逻辑一致性 | 5 | 函数命名、注释和逻辑的一致性 |
| 目录结构 | 4 | 目录命名、结构的一致性 |
| 有意义的注释 | 5 | 关键逻辑有注释说明 |

**代码简单高效（25分）**

| 检查项 | 分值 | 说明 |
|--------|------|------|
| 优先使用标准库 | 10 | 使用标准库提供的能力 |
| TypeScript 类型 | 8 | 避免大量使用 any 类型 |
| 避免冗余 | 4 | 避免无效冗余的写法 |
| 实现方案一致 | 3 | 同类问题使用一致的解决方案 |

**安全规范（15分）**

| 检查项 | 分值 | 说明 |
|--------|------|------|
| XSS 防护 | 5 | 避免 innerHTML/v-html 渲染用户输入 |
| 敏感数据处理 | 4 | 敏感信息不存储在 localStorage，不硬编码密钥 |
| 输入校验 | 3 | 用户输入经过校验和转义 |
| 权限校验 | 3 | 不仅依赖前端进行权限控制 |

**性能规范（10分）**

| 检查项 | 分值 | 说明 |
|--------|------|------|
| 资源清理 | 4 | 及时清理定时器/监听器/订阅 |
| 渲染优化 | 3 | 大列表虚拟滚动、避免不必要渲染 |
| 依赖体积 | 3 | 避免引入过大的依赖包 |

### 加分项（每项最高 10 分）

- **高级安全**：CSP 配置、SRI 校验、依赖漏洞扫描
- **性能优化**：代码分割、图片优化、请求缓存、虚拟滚动
- **代码架构**：模块划分合理、接口设计良好、设计模式使用合理
- **功能难度**：巧妙的算法或思路解决复杂问题
- **稳定性**：异常保护、网络请求优化、兜底策略

### 评分等级

| 分数区间 | 等级 | 结论 |
|---------|------|------|
| 90+ | 优秀 | LGTM |
| 80–89 | 良好 | 需要小修改 |
| 70–79 | 合格 | 需要修改 |
| < 70 | 不合格 | 需要重大修改 |

### 一票否决项

无论评分多高，以下问题必须修复后才能合入：

| 问题 | 规则 ID | 说明 |
|------|---------|------|
| XSS 漏洞 | SEC-001 | 存在明确的跨站脚本攻击风险 |
| 敏感信息泄露 | SEC-010 | 代码中硬编码密钥、凭证等敏感信息 |
| 严重内存泄漏 | PERF-001 | 存在明确的内存泄漏且影响用户体验 |
| 高危依赖漏洞 | SEC-008 | 使用了存在高危漏洞的依赖版本 |

## 五、评审意见撰写规范

### 核心原则

- 对事不对人：评论代码，而不是评论开发者
- 解释为什么：帮助开发者理解原因，传递知识
- 提供建议：给出具体的改进方案
- 保持礼貌：使用建设性的语言

### 评论对比示例

```
错误写法：
"为什么这里你使用了线程，显然并发并没有带来什么好处？"

正确写法：
"这里的并发模型增加了系统的复杂性，但没有实际的性能优势。
建议改为单线程处理，可以降低复杂度并减少潜在的竞态条件风险。"
```

### 评论格式模板

```
[必须/建议/Nit] 规则ID: 问题简述

问题: 具体描述问题所在
原因: 解释为什么这是一个问题
建议: 提供具体的修复方案
参考: 相关规范或文档链接（可选）
```

**示例：**

```
[建议] STYLE-001: 建议使用 const 代替 let

问题: 第 42 行的变量 total 使用了 let 声明
原因: total 在后续代码中没有被重新赋值，使用 let 会误导读者认为该变量可能被修改
建议: 将 let total 改为 const total
```

### 结论类型

| 结论 | 含义 | 行动 |
|------|------|------|
| `LGTM` | Looks Good To Me | 可以合入 |
| `需要小修改` | 有小问题需要处理 | 修复后可合入 |
| `需要修改` | 有明显问题需要修复 | 修复后重新评审 |
| `需要重大修改` | 有设计级别问题 | 需要讨论方案 |
